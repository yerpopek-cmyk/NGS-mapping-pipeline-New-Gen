## 1. Зачем оценивать качество?

```
Проблема: автоматический пайплайн выдаст результат ВСЕГДА,
          даже если данные плохие. Мусор на входе → мусор на выходе.

Что может пойти не так:
  ✗ Контаминация ДНК в RNA-Seq → ложные варианты
  ✗ PCR-дубликаты → переоценка экспрессии генов
  ✗ GC-смещение → неравномерное покрытие → пропущенные варианты
  ✗ Неправильный insert size → ошибки сборки и вариант-коллинга
  ✗ Низкая сложность библиотеки → бессмысленно секвенировать больше

Метрики дают ОБЪЕКТИВНЫЙ ответ, а IGV — визуальное подтверждение.
```

## 2. PCR-дубликаты

### Что это такое

```
Библиотека NGS: фрагменты ДНК с адаптерами → амплификация PCR → секвенирование

ПЦР-дубликат: несколько ридов от ОДНОЙ исходной молекулы
              (а не от разных молекул из того же региона генома)

                 Молекула 1
                    ↓ PCR
           ┌── копия 1 → рид A
           ├── копия 2 → рид B  ← дубликат рида A!
           └── копия 3 → рид C  ← дубликат рида A!

Оптический дубликат: один кластер на flow cell принят за два
                     (ошибка секвенатора, не биологическая)
```

### Почему дубликаты вредны

```
Задача              │ Вред от дубликатов
────────────────────┼──────────────────────────────────────────────
RNA-Seq экспрессия  │ Ген с 1 молекулой кажется высокоэкспрессированным
ChIP-Seq пики       │ Ложные острые пики из артефактов ПЦР
WGS вариант-коллинг │ Ложное увеличение покрытия → выше уверенность
                    │ в гомозиготных вариантах (но не WES с ампликонами!)
ATAC-Seq            │ Ложные пики открытого хроматина

Исключение:
  Ампликонное секвенирование → дубликаты ОЖИДАЕМЫ (все риды из одного ампликона)
  UMI-библиотеки → можно точно различить дубликаты от настоящих молекул
```

### Как Picard определяет дубликаты

```
Алгоритм:
  1. Сортировать риды по позиции 5'-конца
  2. Группировать риды с одинаковыми 5'-концами (у обоих ридов пары)
  3. В каждой группе: один рид с наивысшим суммарным качеством → оригинал
                      остальные → дубликаты (помечаем флагом 0x400)

Оптические дубликаты:
  Picard смотрит на координаты кластеров на flow cell (поле в QNAME)
  Если два кластера слишком близко → оптический дубликат
  Параметр: OPTICAL_DUPLICATE_PIXEL_DISTANCE=2500 (для NovaSeq)
```

### Маркировка vs удаление

```bash
# МАРКИРОВКА (предпочтительно): флаг 0x400 ставится, риды остаются
picard MarkDuplicates \
  I=sorted.bam \
  O=markdup.bam \
  M=dup_metrics.txt \
  OPTICAL_DUPLICATE_PIXEL_DISTANCE=2500   # для patterned flow cell (NovaSeq)

# Пример вывода dup_metrics.txt:
# ESTIMATED_LIBRARY_SIZE  PERCENT_DUPLICATION
# 15,234,000              0.142  ← 14.2% дубликатов — нормально для WGS

# Хорошие значения:
#   WGS/WES: < 20%
#   ChIP-Seq: < 30%
#   RNA-Seq: < 50% (зависит от сложности транскриптома)

# ФИЗИЧЕСКОЕ УДАЛЕНИЕ (когда инструмент не понимает флаги):
samtools view -h -b -F 1024 markdup.bam > dedup.bam
# -F 1024 = исключить риды с флагом DUP (0x400)
```

## 3. Покрытие

### Формула и интерпретация

```
C = (N × L) / G

  C — среднее покрытие (×)
  N — число ридов
  L — средняя длина рида (п.н.)
  G — размер исследуемого региона (п.н.)

Примеры:
  E. coli WGS:    C = (5,000,000 × 100) / 4,640,000 ≈ 108×
  Человек WGS30×: C = (900,000,000 × 100) / 3,200,000,000 ≈ 28×
  WES 100×:       C = (300,000,000 × 100) / 300,000,000 ≈ 100×

Рекомендуемое покрытие:
  Задача                        │ Минимум │ Оптимум
  ──────────────────────────────┼─────────┼────────
  WGS germline варианты         │ 20×     │ 30×
  WGS соматические варианты     │ 40×     │ 80×
  WES                           │ 50×     │ 100×
  ChIP-Seq                      │ 20M ридов│ 40M ридов
  RNA-Seq (дифф. экспрессия)    │ 20M ридов│ 50M ридов
```

### mosdepth — быстрый подсчёт покрытия

```bash
# Базовое использование (работает на markdup.bam)
mosdepth \
  -n \           # не выводить per-base файл (быстрее)
  --fast-mode \  # пропустить дубликаты и secondary
  -F 1024 \      # исключить флаг DUP
  ecoli \        # префикс выходных файлов
  markdup.bam

# Выходные файлы:
#   ecoli.mosdepth.summary.txt   — среднее покрытие по хромосомам
#   ecoli.mosdepth.global.dist.txt — распределение покрытия

# С ограничением по регионам (например, только экзоны):
mosdepth -n --fast-mode --by exons.bed ecoli markdup.bam

# Результат ecoli.mosdepth.summary.txt:
# chrom     length    bases     mean  min  max
# U00096.3  4641652   501298416  108.0  0    312
```

### Равномерность покрытия

```bash
# Позиции с нулевым покрытием:
samtools depth -a markdup.bam | awk '$3==0' | wc -l

# Доля генома с покрытием ≥ 10×:
samtools depth -a markdup.bam | awk '$3>=10' | wc -l | \
  xargs -I{} awk "BEGIN {printf \"%.2f%%\n\", {}*100/4641652}"

# Гистограмма покрытия (bedtools):
bedtools genomecov -ibam markdup.bam -bg > coverage.bedgraph
```

## 4. Insert size

### Что такое insert size

```
Paired-end библиотека:
                  ←—————————— insert ————————————→
  ════════════R1══════════           ════════════R2══════════
  5'──────────────────────────────────────────────────────3'

Insert size = расстояние между 5'-концами R1 и R2
              (включает оба рида + пространство между ними)

TLEN в SAM = insert size (со знаком: положительный для R1, отрицательный для R2)

Типичные значения:
  WGS/WES:    150-400 п.н.
  ATAC-Seq:   пики при 200, 400, 600 (нуклеосомы!)
  Hi-C:       распределение > 1000 п.н.
  Mate-pair:  2000-10000 п.н. (ДРУГАЯ ориентация ридов: RF вместо FR!)
```

### Оценка Picard

```bash
picard CollectInsertSizeMetrics \
  I=markdup.bam \
  O=insert_size_metrics.txt \
  H=insert_size_histogram.pdf \
  M=0.5    # минимальная доля ридов для анализа

# Ключевые поля в insert_size_metrics.txt:
# MEDIAN_INSERT_SIZE   MEAN_INSERT_SIZE   STANDARD_DEVIATION
# 286                  291.3              52.1
```

### Ориентации ридов

```
Стандартная Illumina (FR = Forward-Reverse):
  R1: ═══════►           (плюс-цепь)
  R2:        ◄═══════    (минус-цепь)
  TLEN: положительный для R1

Mate-pair (RF = Reverse-Forward):
  R1: ◄═══════           (минус-цепь)
  R2:        ═══════►    (плюс-цепь)
  TLEN: отрицательный для R1

Если библиотека FR, но вы говорите выравнивателю RF → неверное картирование!
HISAT2 параметр: --rna-strandness RF / FR / R / F
```

## 5. GC-смещение

### Механизм возникновения

```
Причина: ПЦР-амплификация неравномерна для разных GC-составов
  GC-богатые фрагменты (>70% GC):
    → Вторичные структуры ДНК → полимераза «застревает»
    → Недоамплифицируются → низкое покрытие

  AT-богатые фрагменты (<30% GC):
    → Денатурируются легче → реамплифицируются
    → Переамплифицируются → высокое покрытие

Итог: покрытие зависит от состава нуклеотидов, а не только от биологии
```

### Оценка

```bash
picard CollectGcBiasMetrics \
  I=markdup.bam \
  O=gc_bias_metrics.txt \
  CHART=gc_bias_chart.pdf \
  S=gc_summary.txt \
  R=reference.fasta

# Ключевые поля:
# GC   WINDOWS  READ_STARTS  MEAN_BASE_QUALITY  NORMALIZED_COVERAGE  ERROR_BAR_WIDTH
# 0    156      0            0.0                0.0                   0.0
# 50   10234    11200        35.2               1.09                  0.02
# 80   2341     1890         32.1               0.81                  0.04
# ↑                                             ↑ идеально = 1.0

# Нормализованное покрытие:
#   ≈ 1.0 для всех GC → всё хорошо
#   0.5 при GC>70% → сильное смещение (проблема с ПЦР)
```

## 6. Кривая насыщения библиотеки

### Зачем нужна

```
Вопрос: если мы будем секвенировать больше, получим ли новые молекулы?
        Или только копии уже виденного (дубликаты)?

Сложная библиотека:    каждый новый рид скорее всего новая молекула
Простая библиотека:    больше ридов = больше дубликатов одних и тех же

Применение:
  → Решить, нужно ли досеквенирование
  → Оценить эффективность библиотеки
  → Обнаружить биологические ампликоны (все молекулы из одного локуса)
```

### preseq

```bash
preseq lc_extrap \
  -pe \           # paired-end
  -seg_len 100000 \
  -B markdup.bam \  # -B = BAM-файл (не текстовый)
  -o complexity_curve.txt

# Вывод: таблица (N ридов, ожидаемые уникальные молекулы)
# 0         0
# 500000    487000   ← почти все уникальны (хорошая библиотека)
# 1000000   950000
# 5000000   3800000  ← плато начинается → смысла секвенировать больше меньше

# Визуализация: включается в MultiQC автоматически
```

## 7. Мультикартирование

### Primary vs Secondary vs Supplementary

```
Один рид → несколько строк в BAM:

Primary alignment (FLAG & 0x100 = 0):
  Лучшее выравнивание. Только одно на рид.
  MAPQ = реальное значение.

Secondary alignment (FLAG & 0x100 = 1):
  Альтернативное место картирования (≥ 2-е лучшее)
  MAPQ = 0 (неопределённость)
  → обычно фильтруем: samtools view -F 256

Supplementary alignment (FLAG & 0x800 = 1):
  Химерный рид: часть рида выровнена на другой хромосоме
  → Транслокации, инверсии, длинные вставки
  → ОСТАВЛЯЕМ для поиска структурных вариантов

Мультикартированные риды (MAPQ = 0):
  → Повторяющиеся регионы: LINE, SINE, LTR, центромеры, теломеры
  → Паралогичные гены: olfactory receptors, HLA
```

### Стратегии для разных задач

```bash
# WGS/WES вариант-коллинг: убрать secondary, дупликаты
samtools view -b -F 1796 -q 20 markdup.bam > for_variant_calling.bam
# 1796 = 4+8+256+512+1024

# RNA-Seq подсчёт экспрессии: только unique
samtools view -b -q 20 markdup.bam > unique_mapped.bam

# ChIP-Seq (strict): только уникальные
samtools view -b -q 30 markdup.bam > chipseq.bam

# Для изучения repeat regions: ОСТАВЛЯЕМ MAPQ=0
samtools view -b markdup.bam > all_aligned.bam
# Но используем RepeatMasker-аннотацию для интерпретации
```

## 8. Специфика RNA-Seq

### Распределение ридов по частям гена

```
Для poly-A RNA-Seq (стандартная библиотека):
  Ожидаем:
    CDS_Exons:   ~60-70% ридов
    5'UTR:       ~5-10%
    3'UTR:       ~15-20%
    Introns:     ~5-10%  (pre-mRNA / геномная контаминация)
    Intergenic:  ~1-5%

  Если Introns + Intergenic >> 20% → ДНК-контаминация или деградация РНК

  Инструмент: read_distribution.py (RSeQC)
  Требует BED-файл аннотации (не GTF!)
```

### Конвертация GTF → BED для RSeQC

```bash
# GTF → GenePred → BED
gtfToGenePred \
  -genePredExt \
  -ignoreGroupsWithoutExons \
  annotation.gtf annotation.genePred

genePredToBed annotation.genePred annotation.bed

# Теперь можно использовать:
read_distribution.py \
  -r annotation.bed \
  -i dedup.bam \
  > read_distribution.txt

geneBody_coverage.py \
  -r annotation.bed \
  -i dedup.bam \
  -o genebody_coverage
# Показывает: равномерно ли покрыты тела генов от 5' к 3'?
# 3' bias → деградация РНК или poly-A enrichment
```

### Read groups — почему важны

```
Read Group (@RG в заголовке SAM) = метаданные о происхождении рида.
  RGID: уникальный ID (обычно = SRA accession)
  RGSM: имя образца
  RGLB: название библиотеки
  RGPL: платформа (ILLUMINA)
  RGPU: unit (номер lane/flow cell)

Зачем:
  GATK HaplotypeCaller использует RGSM для группировки образцов
  Можно объединить несколько lane → один BAM → GATK анализирует вместе
  Без RG → GATK откажется работать!

Добавить RG к существующему BAM:
  picard AddOrReplaceReadGroups \
    I=input.bam O=with_rg.bam \
    RGID=SRR36689450 RGLB=lib1 RGPL=ILLUMINA RGPU=unit1 RGSM=yeast_sample
```

## 9. Чеклист QC

### Для WGS/WES

| Метрика | Инструмент | Хорошо | Плохо |
|---|---|---|---|
| % картировалось | samtools flagstat | > 95% | < 90% |
| % properly paired | samtools flagstat | > 90% | < 80% |
| % дубликатов | Picard MarkDuplicates | < 20% | > 30% |
| Среднее покрытие | mosdepth | 30× (WGS) | < 15× |
| % геном ≥ 10× | mosdepth | > 90% | < 80% |
| Median insert size | Picard InsertSize | 200-400 п.н. | < 100 или > 800 |
| GC normalized cov. | Picard GcBias | ~1.0 для всех GC | < 0.5 при GC > 70% |

### Для RNA-Seq

| Метрика | Инструмент | Хорошо | Плохо |
|---|---|---|---|
| % картировалось | samtools flagstat | > 85% | < 70% |
| % уникально | samtools flagstat (MAPQ≥20) | > 70% | < 50% |
| % дубликатов | Picard MarkDuplicates | < 50% | > 70% |
| % ридов на экзоны | RSeQC read_distribution | > 60% | < 40% |
| % ридов на интроны | RSeQC read_distribution | < 15% | > 30% |
| 3'-bias (gene body) | RSeQC geneBody_coverage | равномерно | сильный 3' перекос |

### Команды для быстрого чеклиста

```bash
# Всё сразу → один HTML через MultiQC
multiqc . -o multiqc_report/

# Быстрая проверка флагов:
samtools flagstat markdup.bam

# Быстрая проверка покрытия:
samtools coverage markdup.bam | column -t

# Qualimap — всё в одном (WGS):
qualimap bamqc \
  -bam markdup.bam \
  -gff annotation.gff \
  -outdir qualimap_out \
  --java-mem-size=4G \
  -c   # цветные графики

# Qualimap для RNA-Seq:
qualimap rnaseq \
  -bam markdup.bam \
  -gtf annotation.gtf \
  -outdir qualimap_rnaseq
```
