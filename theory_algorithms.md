## 1. Задача поиска подстроки

Картирование — это в основе своей задача нахождения N-миллионных коротких строк (ридов) внутри одной длинной строки (геном ~3 млрд букв).

```
Геном:  ATGCATGCATGCTAGCTAGC...  (длина N)
Рид:    ATGCTAGC                 (длина M)
Задача: найти все вхождения рида в геноме
```

Наивный подход — идти по геному позицию за позицией и сравнивать.  
**Проблема**: миллиард ридов × миллиарды позиций = невозможно.

## 2. От линейного к FM-индексу

| Алгоритм | Сложность поиска | RAM (геном человека) | Используется? |
|---|---|---|---|
| Линейный поиск | O(N × M) | O(N) ~3 GB | ✗ слишком медленно |
| Бинарный (Suffix Array) | O(M × log N) | O(N) ~12 GB | частично |
| Суффиксное дерево | O(M) | ~64 GB | ✗ слишком много RAM |
| **BWT + FM-index** | **O(M)** | **~3-4 GB** | **✓ везде** |

> Ключевая идея: один раз потратить время на подготовку (индексирование), чтобы потом искать миллиарды ридов быстро.

### Суффиксный массив — интуиция

```
Строка: BANANA$   ($ — терминатор, лексически меньше всех)

Суффиксы, отсортированные лексикографически:
  позиция 6: $
  позиция 5: A$
  позиция 3: ANA$
  позиция 1: ANANA$
  позиция 0: BANANA$
  позиция 4: NA$
  позиция 2: NANA$

Бинарный поиск по этому массиву: O(M × log N)
Хранить нужно только позиции [6,5,3,1,0,4,2] — O(N) памяти
```

## 3. BWT в деталях

### Построение матрицы ротаций

```
Строка: BANANA$

Шаг 1 — все циклические ротации:
  BANANA$
  ANANA$B
  NANA$BA
  ANA$BAN
  NA$BANA
  A$BANAN
  $BANANA

Шаг 2 — сортируем лексикографически:
  $BANANA     ← строка 0
  A$BANAN     ← строка 1
  ANA$BAN     ← строка 2
  ANANA$B     ← строка 3
  BANANA$     ← строка 4
  NA$BANA     ← строка 5
  NANA$BA     ← строка 6

Шаг 3 — запоминаем только F (первый столбец) и L (последний):

  F = [ $, A, A, A, B, N, N ]
  L = [ A, N, N, B, $, A, A ]
       ↑
       Это и есть BWT(BANANA$) = "ANNB$AA"
```

### Ключевое свойство BWT

```
L[i] — это символ, стоящий в исходной строке ПЕРЕД символом F[i].

Почему? Потому что каждая строка матрицы — это циклический сдвиг.
Последний символ строки = символ, стоящий перед первым в оригинале.

Это свойство называется LF-mapping:
  L[i] ──── соответствует ──── F[?]
  (тот же символ, та же его порядковая позиция в геноме)
```

### Поиск паттерна через FM-index

Ищем `P = ANA` в BWT(BANANA$):

```
Структуры FM-index:
  F = [$, A, A, A, B, N, N]
  L = [A, N, N, B, $, A, A]
  
  Count[c]   = число символов < c в F (предподсчитано):
    Count[$] = 0
    Count[A] = 1
    Count[B] = 4
    Count[N] = 5
  
  Occ[c][i]  = число вхождений символа c в L[0..i] (предподсчитано)

──────────────────────────────────────────────────────
Шаг 1: последняя буква паттерна 'A' → найти все строки F начинающиеся с A
  Count[A] = 1  → диапазон в F: строки [1, 3]  (три буквы A)

Шаг 2: смотрим L[1..3] = [N, N, B], ищем предпоследнюю букву 'N'
  L[1] = N ✓  L[2] = N ✓  L[3] = B ✗
  → строки 1 и 2 — кандидаты

Шаг 3: LF-mapping — прыгаем в F для строк с N
  Count[N] = 5, Occ[N][1] = 1, Occ[N][2] = 2
  → в F это строки [5, 6]  (буквы N на позициях 5 и 6)

Шаг 4: смотрим L[5..6] = [A, A], ищем первую букву 'A'
  L[5] = A ✓  L[6] = A ✓
  → диапазон не сужается: [5, 6]

Результат: паттерн ANA найден в 2 местах!
  Из суффиксного массива: SA[5]=1, SA[6]=3
  → позиции 1 и 3 в строке BANANA ✓
──────────────────────────────────────────────────────

Сложность: O(|P|) = O(3) — не зависит от длины генома!
```

### Почему FM-index эффективен по памяти

```
Суффиксное дерево: нужно хранить указатели на все узлы → ~20 байт × N = 64 GB
FM-index:
  - BWT строка L: 2 бита/нуклеотид × 3×10⁹ = ~750 MB
  - Count[]: 4 числа
  - Occ[][]: хранится сжато (wavelet tree) → ~1-2 GB
  Итого: ~3 GB для генома человека ✓
```

## 4. Алгоритмы выравнивания

FM-index находит **точные** совпадения. Но реальные риды содержат ошибки и мутации — нужны алгоритмы, допускающие различия.

### Весовая функция выравнивания

```
W = Σ score(позиция)

score(match)           = +1  (или +2)
score(mismatch)        = −1  (или −3)
score(gap открытие)    = −5
score(gap продление)   = −1  (дешевле открытия!)

Почему gap open > gap extend?
  Биологически: одна делеция на 5 п.н. = одно событие.
  Пять независимых делеций по 1 п.н. = маловероятно.
  → Affine gap penalty моделирует реальную биологию.
```

### Нидлман-Вунш (глобальное выравнивание)

```
Идея: динамическое программирование.
Матрица DP[i][j] = лучший вес выравнивания
                   первых i символов seq1 с первыми j символами seq2.

Рекуррентность:
  DP[i][j] = max(
    DP[i-1][j-1] + score(A[i], B[j]),   ← совпадение/замена
    DP[i-1][j]   + gap_penalty,          ← гэп в seq2
    DP[i][j-1]   + gap_penalty           ← гэп в seq1
  )

Сложность: O(M × N) — медленно для больших последовательностей!

Пример: GAGAC vs GAAT (match=+1, mismatch=−1, gap=−2)

      ε   G   A   A   T
  ε   0  −2  −4  −6  −8
  G  −2   1  −1  −3  −5
  A  −4  −1   2   0  −2
  G  −6  −3   0   1  −1
  A  −8  −5  −2   1   0
  C −10  −7  −4  −1   0

Оптимальное выравнивание (traceback):
  GAGAC
  GA-AT
  Score = 0
```

### Смит-Вотерман (локальное выравнивание)

```
Отличия от Нидлмана-Вунша:
  1. DP[i][j] = max(рекуррентность, 0) — никогда не отрицательный
  2. Traceback начинается с МАКСИМАЛЬНОГО элемента матрицы
  3. Traceback останавливается при встрече нуля

Когда использовать:
  → Когда рид выровнен лишь на часть референса
  → Структурные варианты (химерные риды)
  → Поиск гомологичных белковых доменов

В картировщиках (BWA-MEM2) используется ускоренная версия
  через SSE2/AVX2 инструкции (в 10-20 раз быстрее naive)
```

### Сравнение глобального и локального

```
Последовательность 1: GCGACTGA
Последовательность 2: GAGAT

Глобальное (Нидлман-Вунш):
  GCG-ACTGA
  -GAGAT---
  Штрафуем за все гэпы, включая концевые

Локальное (Смит-Вотерман):
  GCGACT
    GAGA T
  Штрафуем только за внутренние гэпы → лучший локальный участок
```

## 5. Seed-and-extend: как работает BWA-MEM2

FM-index = точный поиск. Smith-Waterman = медленный, но точный.  
**Решение**: комбинировать оба.

```
Алгоритм seed-and-extend:

┌─────────────────────────────────────────────────────────────┐
│ ШАГ 1 — SEED (точный поиск через FM-index)                  │
│                                                             │
│   Рид: ATGCTAGCATGCTAGC                                     │
│              ↓                                              │
│   Берём короткий k-мер (seed, ~19 п.н.): ATGCTAGC          │
│   Ищем через FM-index за O(k) → список позиций в геноме    │
│   Несколько seed'ов из одного рида → несколько кандидатов  │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ ШАГ 2 — EXTEND (локальное выравнивание вокруг seed'а)       │
│                                                             │
│   Референс: ...XXXXXATGCTAGCATGCTAGCXXXXX...               │
│                      ↑ seed нашли здесь                    │
│   Расширяем Smith-Waterman влево и вправо                  │
│   Допускаем мисматчи и гэпы                                │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ ШАГ 3 — ВЫБОР лучшего выравнивания                         │
│                                                             │
│   Берём позицию с максимальным AS (alignment score)        │
│   MAPQ = f(AS_best, AS_second_best)                        │
│   Чем ближе AS первого и второго → тем ниже MAPQ           │
└─────────────────────────────────────────────────────────────┘
```

### Soft clipping

```
Концы ридов часто содержат адаптеры или имеют низкое качество.
BWA-MEM2 с флагом -Y не выкидывает их (hard clip),
а помечает как soft-clipped в CIGAR:

  Рид:         AAAA_GCTAGCATGC_TTTT
  Адаптер слева: AAAA (не выравнивается)
  Адаптер справа: TTTT (не выравнивается)

  CIGAR: 4S12M4S
         ↑        ↑
         soft-    soft-
         clipped  clipped

Нуклеотиды сохраняются в поле SEQ → не теряем данные!
Это важно для поиска структурных вариантов по soft-clipped ридам.
```

## 6. Картирование РНК: STAR vs HISAT2

### Проблема сплайсинга

```
ДНК:  ──[ЭКЗОН 1: 200 п.н.]──[ИНТРОН: 1500 п.н.]──[ЭКЗОН 2: 150 п.н.]──
                                     ↓ сплайсинг
мРНК: ──[ЭКЗОН 1]──[ЭКЗОН 2]──

Рид из RNA-Seq перекрывает границу экзон1-экзон2:
  [рид 100 п.н.: ════════════════════]
  [экзон1=====][       интрон        ][=====экзон2]

BWA-MEM2 ищет непрерывный участок → не найдёт!
CIGAR для такого рида: 50M1500N50M  (N = intron skip)
```

### STAR vs HISAT2

| Параметр | STAR | HISAT2 |
|---|---|---|
| Алгоритм индекса | Суффиксный массив | BWT + FM-index |
| RAM | 30-40 GB | 4-8 GB |
| Скорость | Очень быстро | Быстро |
| Требует GTF? | Да (для индекса) | Нет (опционально) |
| Новые сплайс-сайты | Да | Да |
| Немодельные виды | Сложнее | Предпочтителен |

```
Когда STAR:    модельный организм (человек, мышь), есть GTF, много RAM
Когда HISAT2:  немодельный вид, нет GTF, мало RAM
```

### Как HISAT2 находит интроны

```
1. Берёт рид, ищет seed в геноме через FM-index
2. Если рид не картируется полностью → пытается разбить на части
3. Ищет второй anchor-seed (оставшиеся нуклеотиды после разрыва)
4. Если два anchor'а соединяются через известный сплайс-сайт
   (GT/AG — канонический интрон) → принимает как сплайсинг
5. CIGAR: Nп.н. = длина интрона
```
